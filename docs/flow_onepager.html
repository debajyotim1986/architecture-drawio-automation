Normalize → technical "surgical" docTerminal

normalize_jira_story.py rewrites the raw note into the canonical 11-section format the builder relies on, using template/surgical-story-template.txt as the reference shape. The original file is never modified.

python scripts/normalize_jira_story.py jira-stories/<KEY>.txt # add --diagram <FILE> for UPDATE (surgical-diff) mode
In → Out jira-stories/<KEY>.txt jira-stories/<KEY>-surgical.txt
⑂

Two modes & a fallback

CREATE (no --diagram): canonical input is copied verbatim; raw input gets an LLM rewrite.  UPDATE (--diagram <FILE>): always LLM-rewritten, embedding read_diagram_summary so the surgical quotes existing labels for explicit REMOVE/ADD.  No API key? Exit 2 emits a NEEDS_NORMALIZATION payload — the LLM writes the surgical file itself.

2

Read the doc & decide create vs updateLLM

The LLM client reads the surgical doc, surveys existing diagrams, and picks a path: build new, or patch the existing one in place.

read_story(name="<KEY>-surgical.txt") · list_diagrams · read_diagram_summary
MCP tools (read-only) read_story · list_diagrams · read_diagram_summary
3

Build the diagram — swimlanes, then nodes, then edgesMCP

create_diagram makes a timestamped file. Then layer first: 4–6 add_container swimlanes (colored layers) BEFORE any node. add_node drops each component with its official gcp_icon. add_edge connects them — auto-numbered steps, parallel fan-out, row-gap waypoint routing, busy-node cards. One call at a time.

create_diagram(jira_key) → add_container ×N → add_node(parent_id, gcp_icon) → add_edge  (opt. add_title)
Output diagrams/<KEY>-<ts>.drawio
🛡️

Hard gate — verify before anything ships

python scripts/verify_mcp.py --diagram <FILE> checks the 13-tool registry, no dangling edges, no duplicate labels, every GCP node uses its icon, every arrowhead renders, and parallel edges are fanned out. Exit 2 = do not ship — fix with update_node / remove_element and re-verify.

4

Generate the connector-detail tableTerminal

generate_connector_details.py walks the diagram's numbered edges and fills a 14-column table (interaction, protocol, auth, data, network, SLA, reliability, observability, compliance, dependencies) via a deterministic GCP-defaults rule engine — optionally refined by Claude. Zero {TBD} in the output.

python scripts/generate_connector_details.py --diagram <FILE> --story jira-stories/<KEY>-surgical.txt
Output jira-stories/<KEY>-connectors.md
5

Embed the table below the diagramMCP

add_connector_table reads the .md and lays a styled table beneath the swimlanes — one row per numbered edge — so opening the file shows the architecture and the per-connector contract on one canvas. A zero-TBD pre-flight guard refuses to embed a half-filled table.

add_connector_table(diagram="<KEY>-<ts>.drawio")
Updated artifact diagrams/<KEY>-<ts>.drawio
✓

Deliver — deck-quality, verified, documented

The result is a layered .drawio with official GCP icons, numbered data-flow connectors, an embedded integration-contract table, and a passing verifier — fully reproducible from the same prompt. Re-runs are timestamped, so prior diagrams are never overwritten.
